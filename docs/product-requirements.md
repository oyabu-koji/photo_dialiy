# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**Photo Diary** - 写真と音声で思い出を残すライフログアプリ

### プロダクトコンセプト
- **Photo-Centered Memory Log**: 写真を起点に、その瞬間の文脈をイベント単位で記録する
- **Voice-First Context Capture**: 長文入力の代わりに短い音声メモで感情や状況を残す
- **Offline-First Family Archive**: 通信状況に左右されず、家族の思い出を端末上で確実に保存する

### プロダクトビジョン
Photo Diary は、写真が大量に残っても「その時どう感じたか」「何があったか」が失われる問題を解消する。  
ユーザーは写真を選び、短い音声を残すだけで、出来事をイベント単位の Memory Log として保存できる。  
MVP ではオフラインで完結する個人用モバイルアプリとして成立させ、将来的には家庭内バックアップや AI 補助によって思い出の再利用価値を高める。  
SNS 共有ではなく、家族や個人が長く振り返るための私的な記録体験を提供する。

### 目的
- 写真だけでは不足する文脈を、音声と位置情報で補完する
- 日常や旅行の記録をイベント単位で整理し、後から振り返りやすくする
- 最小入力で保存できる UX を提供し、記録の継続率を高める
- オフライン前提で、通信がない環境でも確実に保存できる土台を作る

## ターゲットユーザー

### プライマリーペルソナ: 佐藤美咲 (34歳、会社員・2児の母)
- iPhone を使って日常的に子供や旅行の写真を撮る
- 写真アプリには画像が溜まるが、出来事や会話の記憶が残らないことに不満がある
- 長文の日記入力は続かず、短時間で保存できる仕組みを求めている
- 家族の思い出を SNS ではなく私的に保管したい
- 平日は子供の成長記録、休日は旅行や外出イベントをまとめて残したい

## 成功指標(KPI)

### プライマリーKPI
- イベント保存完了率: 80%以上 (ベータ公開後 3 か月以内)
  - 写真選択開始から保存完了まで到達したセッションの割合
- 週次アクティブ記録ユーザー率: 50%以上 (ベータ公開後 3 か月以内)
  - 週に 1 件以上のイベントを保存したユーザーの割合
- 保存イベントあたり音声添付率: 60%以上 (ベータ公開後 3 か月以内)
  - 写真のみではなく音声も活用されているかを測る

### セカンダリーKPI
- 位置情報解決率: 70%以上 (写真 EXIF または端末位置情報で解決)
- 7 日以内の再訪率: 40%以上
- イベント保存失敗率: 2% 未満

## 機能要件

### コア機能(MVP)

#### イベント作成

**ユーザーストーリー**:  
ユーザーとして、思い出をすぐ残すために、複数写真と 1 件の音声メモを 1 つのイベントとして保存したい

**受け入れ条件**:
- [ ] ユーザーはカメラまたはフォトライブラリから 1 枚以上の写真を選択できる
- [ ] ユーザーはイベントに対して 1 件の音声メモを録音、再録音、削除できる
- [ ] 保存時にイベントタイトル、日時、位置情報が自動補完される
- [ ] 保存完了後、イベントはローカル端末に永続化され、アプリ再起動後も参照できる
- [ ] 写真はアプリ内に重複コピーせず、端末上の参照 URI を保持する

**優先度**: P0(必須)

#### イベント一覧タイムライン

**ユーザーストーリー**:  
ユーザーとして、過去の出来事を時系列で振り返るために、保存済みイベントを一覧で見たい

**受け入れ条件**:
- [ ] 一覧にはサムネイル、タイトル、イベント日付、位置名を表示する
- [ ] サムネイルには `photos[0]` を使用する
- [ ] 新しいイベントが先頭に表示される
- [ ] 一覧からイベント詳細画面へ遷移できる
- [ ] データ件数が 200 件規模でも、初回一覧表示が著しく遅延しない

**優先度**: P0(必須)

#### 地図表示

**ユーザーストーリー**:  
ユーザーとして、思い出を場所で振り返るために、イベントを地図上で確認したい

**受け入れ条件**:
- [ ] 位置情報を持つイベントのみを地図上のピンとして表示する
- [ ] ピンには写真サムネイルを紐づけてプレビュー表示できる
- [ ] ピン選択からイベント詳細画面へ遷移できる
- [ ] 位置情報は `写真 EXIF の平均座標 -> AddEntry 時の GPS -> 位置なし` の順で決定する
- [ ] 位置情報がないイベントは地図表示対象外とし、一覧や詳細からは閲覧できる

**優先度**: P0(必須)

#### イベント詳細閲覧と編集導線

**ユーザーストーリー**:  
ユーザーとして、保存した思い出を確認し直したり修正したりするために、写真、音声、位置情報をまとめて見て編集したい

**受け入れ条件**:
- [ ] `EntryDetailScreen` は閲覧専用とし、写真一覧、音声再生、位置名、日付を確認できる
- [ ] `EntryDetailScreen` の `Edit` 操作から `AddEntryScreen` の編集モードへ遷移できる
- [ ] `AddEntryScreen` は新規作成と既存イベント編集の両方を担当する
- [ ] 編集モードではタイトル更新、音声の再録音 / 削除、写真の追加・削除を行える
- [ ] 写真の追加・削除を保存した場合、位置情報が再計算される
- [ ] 編集後の内容はローカル永続化される
- [ ] 作成 / 編集の保存後は `EntryDetailScreen` で結果を確認できる
- [ ] 写真が 0 枚の状態では保存できない

**優先度**: P0(必須)

#### ローカル保存とオフライン継続

**ユーザーストーリー**:  
ユーザーとして、通信がない状況でも記録を失わないために、イベントを端末内で確実に保存したい

**受け入れ条件**:
- [ ] イベントメタデータはローカル DB に保存される
- [ ] 音声ファイルはアプリ専用領域に保存される
- [ ] ネットワーク未接続でもイベント作成、一覧、詳細、地図表示が利用できる
- [ ] 権限拒否や位置情報取得失敗時も、写真と音声中心の保存フローは継続できる

**優先度**: P0(必須)

### 将来的な機能(Post-MVP)

#### ローカルバックアップ同期

Mac mini 上のバックアップサーバーへ、写真、音声、イベントメタデータを非同期同期する。  
同期は Wi-Fi 接続時またはアプリ起動時にトリガーする。

**優先度**: P1(重要)

#### 家族向け共有・閲覧最適化

複数端末からの閲覧、家族単位の分類、アルバム共有などを検討する。

**優先度**: P2(できれば)

#### AI 補助機能

音声からの日記生成、写真や位置からのイベント自動推定、ベストショット選定を検討する。

**優先度**: P2(できれば)

## 非機能要件

### パフォーマンス
- 一覧初回表示: 200 件のイベントを持つ状態で 1.5 秒以内に主要 UI を表示する
- イベント保存: 写真参照情報と音声ファイル保存を含め、通常ケースで 3 秒以内に完了する
- 音声再生開始: ローカル保存済み音声をタップ後 1 秒以内に再生開始する

### ユーザビリティ
- 新規ユーザーが初回起動から 90 秒以内に最初のイベントを保存できる
- 写真選択、音声録音、保存の主要操作は片手操作を意識した配置にする
- 文字入力なしでも最低限の保存が完了する

### 信頼性
- 保存完了表示後にアプリを強制終了しても、保存済みイベントが失われない
- 位置情報取得失敗、音声録音中断、写真権限不足が発生しても、致命的エラーでアプリが終了しない
- イベント保存失敗時はユーザーが再試行可能な状態を保持する

### セキュリティ
- 要求権限は写真、マイク、位置情報のみに限定し、不要な権限を追加しない
- データはユーザー同意なしに外部クラウドへ送信しない
- ログやエラー表示に生のファイルパスや個人情報を露出しない

### スケーラビリティ
- 端末内で 2,000 件程度のイベントと各イベント最大 20 枚の写真参照を扱える設計とする
- 同期機能追加時に `syncStatus` と `syncQueue` を拡張可能なデータモデルとする

## スコープ外

明示的にスコープ外とする項目:
- SNS 共有
- クラウド前提のリアルタイム同期
- AI 日記生成
- AI イベント自動生成
- Web 版アプリ
