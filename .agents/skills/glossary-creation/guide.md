# 用語集作成ガイド

## 目的

`docs/glossary.md` は、PRD、機能設計、アーキテクチャ、リポジトリ構造、開発ガイドラインで使う用語を統一するための source of truth です。  
新しい用語を作るのではなく、既存ドキュメントの用語を整理して揃える意識で作成してください。

## このプロジェクトで優先するカテゴリ

### 1. ドメイン用語

- ゲーム名
- 画面名
- カード、ペア、クリアなどのプレイ用語
- 状態名

### 2. 技術用語

- React Native
- Expo
- JavaScript
- JSDoc
- Vitest
- `@testing-library/react-native`

### 3. アーキテクチャ用語

- UI Layer
- Application Layer
- Domain Layer
- Platform Layer

### 4. 運用用語

- `.steering`
- PRD
- MVP
- component test

## 書き方

各用語に対して最低でも次を入れます。

- 定義
- 本プロジェクトでの使い方
- 関連用語
- 必要なら英語表記

技術用語では次もあると良いです。

- 公式サイト
- このプロジェクトでの用途
- 参照先ドキュメント

## 追加すべきタイミング

- 新しい状態名を導入した時
- 新しい画面名やコンポーネント名を docs に追加した時
- 新しいツールやライブラリを採用した時
- 「同じ意味なのに呼び方が揺れている」と気づいた時

## 例

### ドメイン用語の例

```markdown
### Feedback

**定義**: プレイヤーの操作や結果に応じて返す視覚、音、振動の反応全般。

**本プロジェクトでの使用**: flip / match / mismatch / finish の演出をまとめて指す。

**関連用語**: Match, Haptics
```

### 技術用語の例

```markdown
### Expo

**定義**: React Native アプリの開発・実行を支援するプラットフォーム。

**本プロジェクトでの用途**: モバイル実行基盤、端末機能利用、開発環境統一。

**関連ドキュメント**: `docs/architecture.md`
```

### エラー例の書き方

```javascript
throw new ValidationError('invalid input');
```

## 避けること

- ドキュメントごとに別の名前を付ける
- 実装に存在しない概念を増やす
- 古い名前を残したまま新しい名前を追加する

## 完成チェック

- [ ] `docs/` 全体で使う重要用語が揃っている
- [ ] JavaScript / Expo / テスト基盤の技術用語が入っている
- [ ] 状態名や画面名が機能設計と一致している
- [ ] `.steering` など運用用語も必要に応じて定義されている
